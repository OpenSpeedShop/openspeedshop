Section 1.     Document Purpose and Overview

The purpose of this document is to describe the test plan SGI have created for Open|SpeedShop.  In this test plan document, SGI discuss the different types of testing SGI will perform, as well as the different strategies SGI will implement.  Additionally, SGI will discuss the different tools SGI have created for the purpose of running the tests.  Finally, a schedule for the completion of the various tasks will be laid out in this document.

 

1.1 Objectives of the Test Plan

 

The objectives of the Open|SpeedShop test plan are:

·         To outline testing strategies, test types and testing techniques used for verification of Open|SpeedShop.

·         To layout the design of the test suite.

·         To describe the tools that will be created for testing.

·         To propose milestones for the test suite.

1.2 Open|SpeedShop Project Information

 

The Open Source Parallel Performance Measurement Package Statement of Work document dated May 27, 2004 contains the technology and strategy to be used by SGI to deliver a parallel performance measurement package.  SGI will create a performance analysis tool for the Intel IA64, IA32, EM64T platforms and AMD AMD64 platform, running under the Linux operating system.

SGI will release, primarily under the GPL and LGPL Open Source licenses, the baseline functionality as described in section 2.1 of the above mentioned SOW and will productize an enhanced Pro version.  Support for single system image (SSI) machines, support for clusters (i.e., multiple OS kernels), exclusive and inclusive user time, program counter (PC) sampling, MPI call tracing, input/output tracing, floating point exception tracing and CPU hardware performance counter experiments are the key components of the baseline functionality.  The performance tool will be designed in such a manner that will allow users to easily extend the tool by adding their own experiments.

Another key feature of the performance tool is its usability.  The user interface will be designed for scientists not computer scientists.  To make the tool usable for a greater range of users, the performance tool will provide novice user Wizards, easily understood language in the user interface components, and context sensitive help.  These features and additional features that comprise the baseline version are described in detail in section 2.1 of the SOW.

The design and structure of this project allows for the performance tool described in this document to be ported to several other platforms through a moderate effort.  Since the performance tool will use dynamic instrumentation as one of the primary building blocks, platforms supported by the Dyninst dynamic instrumentation package are ready candidates for porting.

The performance tool will use dynamic probe class library (DPCL) as a building block component for cluster system support.  The use of DPCL will provide a portable means for the performance tool to supply cluster support for platforms supported by the Dyninst dynamic instruction package.  DPCL uses the Dyninst application programming interface (API) to provide its instrumentation capability.

Availability of the new performance tool based on SpeedShop technology on Linux platforms will enable Fortran (77, 90, and 95), C, and C++ programmers to use an advanced performance analysis tool within the Open Source environment.

Section 2.     Test Suite hardware and software requirements

The Open|SpeedShop test suite is designed to run on any of the Open|SpeedShop supported systems that have a Linux operating system and a Python interpreter (version 2.2 or later) installed. The system where the test suite will be run needs to have the compilers that are supported by the Open|SpeedShop tool installed.  The compilers that will be supported by Open|SpeedShop are:

·       Intel C/C++ and Fortran compilers and GNU compilers (GCC starting with version 3.3.2) for the IA32 and IA64 architectures. 

·       Portland Group and GNU compilers (GCC starting with version 3.3.2) for the AMD64 architectures.

Section 3.     Testing Types

The following paragraphs describe the types of tests that comprise the Open|SpeedShop testing plan.

3.1        Unit testing

Unit testing verifies program logic at the sub-component level. Each unit test verifies the functionality of one function.  Unit testing is not intended to verify the correctness of the entire product, rather small key functions.  The developers who are working on the product will be writing unit tests for their specific portions of the project.  The purpose of unit testing is to test each key high impact function or module the developer writes.  Within reasonable limits, SGI will write unit tests for the various sections of the product.  The framework, a wide cross-section of DPCL functionality, the GUI, the command line, the specific experiment collectors, and the specific experiment user interface plugins will have unit tests written for them.

3.2        Integration testing

Integration testing verifies the entire system’s functionality.  The purpose of integration testing is to verify that the combined parts of a product work together and achieve the desired result.  Integration testing is most critical early in the development process as the components are initially being developed.  SGI intend on performing incremental integration testing, during these early stages of development, test the performance tools product components together.  In doing so SGI will write tests to make sure the entire product interacts with the components in the expected manner. Key interfaces that will be verified will be:

·         The interface between the framework library and the user interface

·         The interface between the framework library and the DPCL API

·         The interface between the collector plugins and the framework library

·         The interface between the user interface plugins and the GUI and Interactive CLI

3.3        Acceptance testing

Acceptance testing verifies that Open|SpeedShop needs to meet the customer’s objectives.  Acceptance testing is the final round of testing that is based on the customer’s specifications.  Each quarter, based on what SGI promised to deliver, both SGI and ASCI will determine a set of mutually acceptable scenarios for determining acceptance.  These scenarios will be specified in the quarter prior to the quarter of milestone delivery.  SGI will test Open|SpeedShop against such scenarios to verify that the portion of the Open|SpeedShop product SGI delivers meets the necessary customer specifications.     Customer specifications are described by the Open Source Parallel Performance Measurement Package Statement of Work document dated May 27, 2004 and by the results of usability testing..

Final acceptance testing will have elements of load, concurrency, performance, and big application testing in the acceptance scenarios.  This will be done on SGI platforms as well as on the ASC laboratory systems that Open|SpeedShop supports.  Final acceptance scenarios will be a SGI/ASC mutually agreed upon set of tests.  The final acceptance tests will be agreed upon at least four months prior to the “Deliver MPI call tracing experiment and final components” milestone delivery date.

Each milestone has an associated Appendix to this document.  The first milestone based acceptance scenario covers the deliverables for the "Deliver program counter sampling experiment" milestone (MS-3).

3.4        Regression testing

Regression testing verifies that the fixes/modifications are correct, and no other parts of the system have been affected.  Following any check in of code by a developer working on the Open|SpeedShop project, a suite of regression tests will be run.  These tests will be designed to verify that whatever changes were checked in have moved the project forward and have not created any regressions in existing code.  The regression tests will verify that new source changes that have been added have not broken adversely affected the existing performance tool functionality.  Regression testing is selective retesting of the software system as a whole to verify that any bugs have been fixed, and no new ones have been introduced.   It is used to ensure that newly added features have not created problems with previous versions of the software.   As in the unit testing, there will be regression tests written for each different aspect of the tool.  For each part, the tests will be designed around the APIs put forward by the developer.  This testing will be done using the batch command tool.   Additionally, it should be noted that the regression tests will be run using the compilers specified in the Open Source Parallel Performance Measurement Package Statement of Work document dated May 27, 2004. The compilers that will be supported by Open|SpeedShop are:

·         Intel C/C++ and Fortran compilers and GNU compilers (GCC starting with version 3.3.2) for the IA32 and IA64 architectures. 

·         Portland Group and GNU compilers (GCC starting with version 3.3.2) for the AMD64 architectures.

The tests will be run against the executables that were created using the compilers listed above.

3.4.1 Adding a Regression test

Our test suite can be found in OpenSpeedShop/test. In this directory there are three relevant directories: test_scripts, src, and executables.

The test_scripts directory contains all the scripts used for running the tests and parsing the results. Scripts are named runall or a version of runone-*. The * represents what specific type of experiment the runone script is used for. For example, runone-pcsamp runs and parses the results of a pcsamp experiment. Given the wide range of outputs one can expect from an Open|SpeedShop experiment, there are many different runone-* scripts. Each one is tailor made for a different format of output.

The src directory contains all the different tests, each contained in its own directory. There are four different directories in src: dpcl, dyninst, regression and unit. The dpcl and dyninst directories contain tests that verify if dpcl and dyninst are working. They are useful to run when Open|SpeedShop is initially installed on a system to determine whether or not two of its necessary components are functioning.

OpenSpeedShop/test/src/regression contains the meat of the test suite. All of the regression tests are contained in this directory, and it is how one can test various command line options and experiments. Within regression there are the following directories: bb (which contains all the command line building block command tests, i.e. expEnable, expClose, etc.), hwc (contains tests for the hwc experiment), hwctime (contains tests for the hwctime experiment), inf (contains all the information tests, i.e. all the list -v options), io (contains tests for the io experiment), mpi (contains tests for the mpi experiment), pcsamp (contains tests for the pcsamp experiment), usertime (contains tests for the usertime experiment)and views (contains tests for the various views, i.e. calltrees).

A typical test directory contains a runall, a runone, a .tst file, a Makefile.am and sometimes and .tst.expected file. runall links to a copy in OpenSpeedShop/test/test_scripts. runall is a general script that traverses the directory tree and runs all the tests, produces a report, and mails it out. For each directory below OpenSpeedShop/test/src/regression there should be a runall. runone is a script that is tailor-made for each type of collector, and for some of the different views. All of the runone scripts are identical with the exception of the 'CompareResults' function. Each one differs according to what kind of output is being parsed, and what type of results are being searched for. Some 'CompareResults' functions compare the output from the OpenSpeedShop run to the ratios and functions that are contained in the *.tst.expected file. In other cases, the script contains all the function names or data that the user is looking for. The *.tst file is a simple shell script that contains the list of commands that are being passed to OpenSpeedShop, creates a file called input.script that contains the commands and is then passed to OpenSpeedShop through the batch command (openss -batch).

3.5        Load testing

Load testing is defined, as identifying the peak load conditions at which the program will fail to handle required processing loads within the required time span.  The nature of Open|SpeedShop is that it will be processing large amounts of data on systems with high numbers of processors, both in single system image (SSI) and cluster configurations.  SGI will write tests to verify that when the amount of data gets extremely large and there are high numbers of processors, the performance tool still functions in the desired manner. In addition, SGI will write tests to verify that a reasonably large trace file will be read and the trace data initialized for analysis within a limit of about five (5) minutes.  For these tests SGI may require test cases from the customer, as well as access to a machine on which SGI can run them. To verify that SGI is meeting the customer’s requirements specified by section 3.4.5 “Higher Scale Measurements” in the Open Source Parallel Performance Measurement Package Statement of Work document dated May 27, 2004, SGI will verify Open|SpeedShop works on a 4000 processor IA64 machine.           

3.6        Performance testing

Performance testing is defined as running the performance tool on itself to see how the tool performs.  Open|SpeedShop is a performance tool that analyzes how an executable performs. Given that, SGI intends on running the tool on itself, to make sure that it is not getting caught in any performance pitfalls. Performance testing will be included in the regression testing to verify that with new check-ins, the tool hasn’t slowed.  In future versions of this document, we will include more specific examples of what the performance testing will include, and the criteria upon which it will be performed.

3.7        Concurrency testing

Concurrency testing is defined as the ability to see that the system does not corrupt data when two or more users update data at the same time.  Since there may be a number of different users running Open|SpeedShop at the same time there is the possibility of data corruption.  For this reason, SGI will write tests to make sure that none of the executables used will be adversely affected by multiple invocations of the performance tool. Furthermore, SGI will test to make sure the tool’s performance is not degraded when more than one person is using it.

3.8        Big application testing

Big application testing is defined as running the performance tool on a big application to see how well it performs.  SGI will create a test suite that contains a number of big applications. SGI will supply some of the large applications to be used in this form of testing and the ASC laboratories will provide other large applications.  The applications will be of the size that the customers expect to use themselves. It should be noted that the big application testing would occur using a number of different compilers.  Just as will be done with the regressions testing, the applications will be compiled and tested with all the compilers specified by the Open Source Parallel Performance Measurement Package Statement of Work document dated May 27, 2004. The compilers that will be supported by Open|SpeedShop and used for big application testing are:

·       Intel C/C++ and Fortran compilers and GNU compilers (GCC starting with version 3.3.2) for the IA32 and IA64 architectures. 

·       Portland Group and GNU compilers (GCC starting with version 3.3.2) for the AMD64 architectures.

 

There are a number of big applications SGI will be using to test Open|SpeedShop.  Included in the large application testing category are:

·         MPQC (Massively Parallel Quantum Chemistry Program), which computes properties of atoms and molecules from the first principles using the time independent Schroedinger equations.  MPQC is a very large C++ application, which is available on SourceForge.net. 

·         QMcBeaver is an object-oriented program that performs Quatum Monte Carlo calculations on atoms and molecules.  It is also available on SourceForge.net. 

SGI will also use a number of applications from the ASCI Purple Benchmark Suite.  The three tests SGI will most closely focus on are sPPM, UMT1.2, and IRS. 

·         sPPM solves 3D gas dynamics problems on a uniform Cartesian mash using a simplified version of PPM. SGI will only use the MPI version of the application.

·         UMT1.2 is a 3D, deterministic, multigroup, photon transport code for unstructured meshes.  It contains a C computation kernel called from an MPI based F90 driver. 

·         IRS or Implicit Radiation Solver code solves the radiation transport equation by the flux-limited diffusion approximation using an implicit matrix solution.  The program is written in standard C, uses MPI and is highly scalable. 

The ASCI Purple Benchmark Suite also contains SPHOT, MDCASK, ParBenCCh, AZTEC, SMG2000, SAGE, Presta Stress Test, Mem Stress Test, and I/O Stress Test. SGI will test with the above mentioned benchmarks if time permits.

It should be noted that the big application testing will be run by hand, and will not be automated.  Upon running Open|SpeedShop on an application, we will examine the results by hand, verify that the tool ran successfully and to completion. SGI will verify that the Open|SpeedShop performance tool is able to measure performance on jobs of up to 256 MPI tasks, in such a manner, as to have a startup cost of less than 2 minutes longer than the time to initiate the jobs without instrumentation. Finally, we will visually verify that the results are within a reasonable spectrum of values.

3.9        GUI testing

GUI testing verifies GUI features and elements and compares them to GUI standards and test design.  For our GUI testing there will be a set of hand run scenarios for commonly done GUI tasks.  SGI will create these scenarios in order to test that various components of the GUI are functioning as expected.

3.10    Dyninst testing

The University of Wisconsin and University of Maryland Dyninst team will test the Dyninst component of the Open|SpeedShop tool.  The Dyninst team will run nightly regression tests, that will encompass the processors and platforms that the Open|SpeedShop tool is required to support. The required compilers and platforms are specified by The Open Source Parallel Performance Measurement Package Statement of Work document dated May 27, 2004.  The Dyninst tests are part of the Dyninst release and thus are available to the users of Dyninst.

3.11    DPCL testing

The DPCL cluster software package comes with a set of verification tests.  SGI will run the DPCL tests along with our regular testing.  DPCL tests are available in the DPCL source tree, which is available for download.  SGI will run these tests during Open|SpeedShop development to verify that no regressions have been introduced.  In addition, SGI may add more tests to the DPCL package, if it is found to be necessary to verify any SGI additions to the DPCL package for this project.  Additional DPCL testing will occur during integration testing.  The integration tests for testing the interface between the Open|SpeedShop framework component and DPCL will test the portions of DPCL that are exercised by the Open|SpeedShop framework component.


Section 4.     Testing strategies

This section defines the functional aspects of the feature to be tested, the strategy for carrying out the tests, and techniques for evaluating the test results.  While testing Open|SpeedShop SGI will employ a number of strategies.  SGI will study test results closely to determine the most appropriate delta SGI will use to determine product correctness.  Regression testing will be performed nightly.  SGI will perform big application testing at least monthly and just prior to a release.  While designing and writing tests, SGI will follow the API of component carefully.  SGI will create a sanity test suite for developers to run before checking in code.  This suite will be thorough enough to catch errors, however not prohibitively large.  Integration testing will be performed incrementally throughout the development cycle.  Finally, once the Open|SpeedShop components are integrated and tested for functionally, SGI will conduct performance testing on the Open|SpeedShop tool itself.

Section 5.     Automation Tools

This section describes the automated tools used to perform the tests and verify the test results. The following scripts will be written to run the unit and regression tests and to verify their results:

5.1        “runone”

“runone” is a script that will take a single test, run it with all the necessary options, and store the results to a results file.  “runone” will be used by “runall”,. It is designed not to be called independently.

5.2        “rununit”

“rununit” is the script that will run a single unit test.  The specific mechanics of the “rununit” script will depend on whether or not the developer checks in a list of tests or the expected output.  However, generally speaking “rununit” will run all         the unit tests in a directory, compare the resulting output to the expected output, then write the results to a report file.

5.3        “runall”

“runall” is the top-level script that is designed to run all the tests.  The user can call “runall” in any directory, and the script will perform all the testing in that directory and all the directories below.  “runall” calls “runone” and “rununit” in all the leaf directories below it.  After each call to either “runone” or “rununit”, “runall” appends the individual results report to a report file of it’s own.  Once the script has traversed its entire tree, it mails its report file to the user that called it.

Section 6.     Naming Conventions

The following naming conventions will be used within the test suite:

6.1        Unit tests

The unit tests will be named according to what part of the tool, what sub-part, and then numbered sequentially.  For example, the 40th unit test written for the PC sampling part of the framework will named unit_fw_pc_40.cxx.

6.2        Big application testing

The tests written for the big applications will be named according to the application, then numbered sequentially.  For example, if there is an application named MadeForTesting, the second test written for it will be named ba_MFT_2.py.

6.3        GUI testing

The tests written for the GUI will be named according to the part of the GUI it is testing.  For example, the 5th test written for the array visualizer will be named gui_arrayvis_5.txt.

6.4        Regressions and Sanity Tests

The tests written for the Regression and Sanity suites will be named according to what the tests are checking.  For example, the 8th test written to check for a regression in the profiling interval settings in PC sampling will be named reg_profint_pcsamp_8.py.

Section 7.     Portability

The test suite will be able to run on any system that is running Linux.  The only platform requirements are that the platform must be able to run Linux and that Open|SpeedShop is supported on that platform.  With respect to additional software support, the test machine will need to have a Python interpreter and C, C++ and Fortran compilers installed in order for the test suite to be fully functional.  In order to run a subset of the test suite based on a specific programming language, the test machine will need the compiler that corresponds to said language to be installed on the machine

Section 8.     Exit Criteria

The following paragraphs describe how SGI will determine when each stage of development is complete.  This is based on the following stages:

8.1        Stage 1: Test Process Development

Test Process Development entails the design team and the tester determining what sort of testing system works best for them.  Once that has been determined, as it has in our case, a team considers themselves out of the Test Process Development stage. 

8.2        Stage 2: Test Planning Stage

During the Test Planning Stage, the testing engineer comes up with a plan to follow for designing the test suite.  Once that plan has been created, and approved by the necessary parties, the tool developers can consider themselves out of this stage.  Currently (September 17, 2004), SGI are in the process of developing our Test Plan.

8.3        Stage 3: Test Design Stage

The process of considering the best ways to test the product, and design such means is what is involved in the Test Design Stage.  Once the tester has sufficiently laid out the design for the test suite, they may move on the creating it.  In order to do so, one must decide what kind of scripts they will write for the testing, what these scripts will deliver, as well as what different kinds of tests need to be written.  Test design will continue throughout the development process. As the developers add new features into the product, new tests will be added.

8.4        Stage 4: Test Automation, Execution, and Defect Tracking Stage

Once a sufficient number of tests have been added to the suite, there will be a regular run of the testing scripts.  Reports will be generated for the developers, and any bugs that are found will be added to our bug tracking system.  This process will continue throughout the development of the product.  SGI will investigate an external bug tracking system to be used to report Open|SpeedShop problems.  SGI will report their findings in the next version of this document.

8.5        Stage 5- Evaluation and Improvement

The goal of a test suite is to catch errors before a product is released, and offer to the customer the most bug free product possible.  SGI will evaluate our test suite by how well it meets this goal.  If the tests written catch a sufficient number of bugs’ prior to release, it is doing its job.  However, SGI will constantly look to improve it, and find ways to further improve the product through better testing.
